*opencode_sprint_loop.txt*  Thin Sprint Loop controller client

=============================================================================
CONTENTS                                                         *SprintLoop*

This plugin requires Neovim 0.12 and an installed `sprint-loop` executable.
Call `require("opencode_sprint_loop").setup()` before any action. It is a
launcher/status client only: it does not start OpenCode, own workflow state,
read controller files, inspect Git, or answer questions.

Install with a plugin manager's normal install/update step and its automatic
help-tag generation. If the manager does not generate help tags, run
`:helptags ALL`. For manual installation, place the checkout under a
`pack/*/start` directory and run this command with its concrete path:

>vim
  :helptags ~/.local/share/nvim/site/pack/sprint-loop/start/opencode_sprint_loop.lua/doc
<

After installation, `:help SprintLoop` opens this document.

SETUP                                                            *SprintLoop-setup*

>lua
  require("opencode_sprint_loop").setup({
    sprint_root = function() return vim.fn.getcwd() end,
    server_url = function() return "http://127.0.0.1:4096" end,
    executable = "sprint-loop",
    web_url = "http://127.0.0.1:4096",
  })
<

`sprint_root` and `server_url` are required. All fields accept a string.
Function-valued `sprint_root`, `executable`, and `server_ca_cert` return one
string synchronously and receive no completion callback. Only `server_url` and
`web_url` functions may return synchronously or call `done(value, error)` once.
URL function results remain private for a five-second arbitration window so
delayed return-plus-callback or duplicate completion rejects before launch.
`executable` defaults to `sprint-loop`. `web_url` is used only by
|SprintLoopOpenSession|. `server_ca_cert`, when supplied, resolves to an
absolute readable regular file. Asynchronous libuv stat rejects FIFOs, devices,
and directories before open; open/fstat/close callbacks then verify readability
and the descriptor type. The path is passed only as
SSL_CERT_FILE to `run` and `resume`; browser CA trust is separate.
Setup resolves root and executable for its mandatory initial status observation;
later actions re-resolve every option they need. Resolver lifetimes are owned by
their setup, watcher, or action lifecycle, including arbitrated results still
queued for delivery.
Commands are registered at plugin load. Before successful setup they report
`setup_required`; repeated setup replaces configuration and watcher state while
preserving pending-request notification deduplication for this Neovim process.
An unknown setup field reports fixed `invalid_setup` without displaying its
untrusted name.

COMMANDS                                                        *SprintLoop-commands*

:SprintLoopStart        Detached `sprint-loop run` launch.        *SprintLoopStart*
:SprintLoopProgress     Show validated controller JSON.        *SprintLoopProgress*
:SprintLoopPause        Delegate `sprint-loop pause`.              *SprintLoopPause*
:SprintLoopResume       Delegate `sprint-loop resume`.            *SprintLoopResume*
:SprintLoopStop         Delegate `sprint-loop stop`.                  *SprintLoopStop*
:SprintLoopOpenSession  Open the active session route.      *SprintLoopOpenSession*

The equivalent public methods are `start`, `progress`, `pause`, `resume`,
`stop`, and `open_session`. They are asynchronous and intentionally do not
return a stable process/status/workflow result. The progress window is
read-only; buffer-local `q` and `Esc` close it.

Before `run` or `resume` argv is constructed, the complete `server_url` must be
a credential-free HTTP(S) origin. Malformed authorities, user-info, paths,
queries, fragments, named-value/provider-token components, and unsupported
schemes are rejected without echoing the value. Browser bases receive the same
complete-value credential scan. Non-detached process stdout and stderr are
bounded during asynchronous streaming.
External stderr is not copied into notifications; non-zero exits and non-zero
process signals use a generic diagnostic directing the user to progress.
Detached `run` stdout/stderr use `/dev/null`, not Neovim-owned pipes. A zero
process exit gets a distinct notice directing the user to progress and never
claims that workflow state is terminal. The asynchronous `/dev/null` open and
queued spawn are generation- and exit-gated. Setup replacement or |VimLeavePre|
closes the pending descriptor without a stale spawn, watcher mutation, or launch
notice; a controller that already spawned successfully is never signalled.

WATCHER                                                         *SprintLoop-watcher*

Setup performs one asynchronous status observation, notifies for a pending
request in that validated document, and then starts the single two-second,
non-overlapping watcher only if a running controller is found.
Successful start/resume also begins launch discovery. A future
`waiting_for_user` projection produces one notification
per request ID directing the user to |SprintLoopOpenSession|. It does not expose
question content or submit an answer. Sprint 2's probe is non-interactive;
waiting is a controlled-fixture compatibility demonstration until Sprint 4.
The watcher performs a final status observation when a launched controller
exits before active state is observed and closes its timer when observation ends.
All plugin status requests use one serialized child slot. Setup replacement and
|VimLeavePre| invalidate the replaced configuration. Watcher and successful
start/resume replacement cancel only setup/watcher reads; public progress and
session reads remain queued and complete normally. Stop keeps the current
watcher across resolver, spawn, command, and success-while-active outcomes until
status confirms `process_running: false`. Controller children are never
signalled by observation replacement.

PROGRESS                                                        *SprintLoop-progress*

The float includes audit remaining effort, CI commit SHA, all checklist counts
including `not_evaluated`, checklist assessment time, and the last event
sequence. Relevant null evidence is shown as `-` rather than omitted.
No-run includes root, `process_running: false`, and controller version. A pre-CI
round above its configured maximum is inconsistent. Final composed lines receive
another credential scan; an unsafe cross-field join is replaced by fixed text.
An interrupted durable active invocation may display `running` while
`process_running` is false; this does not claim that the controller is alive.
Blocked, failed, and stopped projections require a reason.
Persisted status requires the exact V1 state vocabulary, non-null update and
last-event evidence, object-shaped local and pushed commit maps with the same
repository keys, and no active invocation in a terminal state. A
truthful interrupted active invocation in a non-terminal state remains valid.
Recognizable credentials, credential-bearing URLs, URL query/fragment data, and
common provider-token forms in any rendered field reject the complete status as
`inconsistent_status`; the plugin does not render a lossy sanitized projection.
The plugin and controller use the same ASCII case-folding and ASCII-whitespace
credential grammar. Conventional ASCII secrets reject; Unicode long-s,
Kelvin-sign, and NBSP syntax lookalikes are unsupported near misses.

SECURITY                                                        *SprintLoop-security*

Arguments are direct arrays, never shell strings. URLs must be credential-free.
The plugin retains no persistent status/configuration, does not copy complete
environments or CA contents, and uses status JSON as its sole workflow source.
For mkchad, write a configuration callback that reads the existing URL only;
never call a server `ensure`/start function. Do not test against the live
~/.config/mkchad checkout. Use a disposable clone with isolated XDG roots.
The current adapter shape is `vim.g.opencode_opts.server.url(done)` for the
existing URL and `vim.g.opencode_opts.server.ca_cert()` for the CA path; these
belong in user setup and are not hard-coded by the plugin.
Browser path prefixes accept RFC 3986 path characters and valid percent
escapes; raw spaces, malformed escapes, backslashes, and invalid URL characters
are rejected.
Neovim 0.12 browser-handler completion is observed asynchronously with bounded
timer polling. `SystemObj:wait()` force-kills after any elapsed supplied timeout,
including zero, so the plugin never calls it. It non-destructively reads the
result Neovim retains after process exit and inherited output pipes close, until
that result appears or the five-second observation bound expires. Timeout or
observation cancellation never signals or otherwise changes the handler process.
Zero exit is reported as success; non-zero, signalled, or timed-out completion
reports `browser_open_failed`. Missing handlers fail immediately. Unobservable overridden
handlers receive a warning and no terminal-success claim.

CONTROLS                                                        *SprintLoop-controls*

The current Sprint 2 controller responds `feature_not_implemented` for pause,
resume, and stop. The plugin reports a generic `controller_command_failed`
without copying external stderr and makes no local state transition. Recovery
behavior is not implemented here.

ERRORS                                                          *SprintLoop-errors*

Expected failures are reported as concise notifications, including
`setup_required`, `invalid_setup`, `resolver_failed`,
`invalid_resolved_value`, `invalid_server_ca_cert`,
`process_spawn_failed`, `controller_command_failed`,
`status_output_too_large`, `invalid_status_json`,
`unsupported_status_schema`, `inconsistent_status`, `web_url_unavailable`,
`invalid_web_url`, `active_session_unavailable`, `browser_open_failed`, and
`unsupported_neovim`.

The client requires the complete Sprint 3 schema-version-one status projection.
An unsupported or inconsistent projection fails closed; update controller and
plugin together when their compatibility contracts differ. Desktop notification
and URL-handler availability remain properties of the operator environment.

REAL DEMONSTRATION                                      *SprintLoop-real-demo*

The opt-in real procedure is intentionally not part of default verification.
Use a disposable clean sprint-history repository, disposable controller venv,
externally started supported server, synthetic inherited password, optional
disposable regular-file CA, and isolated XDG roots. Never use or inspect the
live ~/.config/mkchad runtime. Put this checkout on `runtimepath`, configure the
explicit executable/root/server/web/CA values, then run an isolated Neovim with
an outer bound from a unique root created by `mktemp -d` such as:

>sh
  timeout 15m env \
    XDG_CONFIG_HOME="$SPRINT_LOOP_DEMO_TEMP/config" \
    XDG_STATE_HOME="$SPRINT_LOOP_DEMO_TEMP/state" \
    XDG_DATA_HOME="$SPRINT_LOOP_DEMO_TEMP/data" \
    XDG_CACHE_HOME="$SPRINT_LOOP_DEMO_TEMP/cache" \
    nvim --clean -u "$SPRINT_LOOP_DEMO_TEMP/init.lua"
<

Run |SprintLoopStart|, |SprintLoopProgress|, and |SprintLoopOpenSession|. Close
and reopen that isolated Neovim to observe setup-time rediscovery. The current
probe ends at `blocked/execution_not_implemented`; waiting remains fixture-only.
Stop the external server yourself, unset synthetic credentials, and remove only
the disposable roots. See the repository README for the complete reproducible
environment and init-file example. This procedure does not claim that an
external OpenCode, browser, or private-CA demonstration has been performed.
